# Predicta KP And Nadi Predicta Strict Phase

This document defines the strict rebuild phase for KP Predicta and Nadi
Predicta across web, mobile, chat, reports, PDF output, and Predicta memory
handoff.

KP and Nadi already have useful foundations, but they must stop feeling like
technical method panels. KP must feel like a precise event-answer room. Nadi
must feel like a clear karmic story room. Both must make users feel that
Predicta understands the question, the context, and the correct school.

## Phase Name

`PREDICTA_KP_NADI_PREDICTA_PHASE_1_EVENT_ANSWER_AND_KARMIC_STORY_ROOM_REBUILD`

## Relationship To Existing Phase Files

This phase builds on top of existing approved contracts. It must not replace,
rename, or collide with them.

  rebuild contract.
- `PREDICTA_REPORT_PDF_PHASE_1A_SCHOOL_REPORT_LANES_AND_MARKETPLACE_SEPARATION`
  owns report marketplace separation. This KP/Nadi phase only defines the
  KP/Nadi lane experience and content quality.
- `PREDICTA_REPORT_PDF_PHASE_7_PREDICTA_MEMORY_AND_CHAT_AWARENESS` owns the
  app-wide Predicta memory/context architecture. This KP/Nadi phase defines the
  KP/Nadi intelligence payloads that must flow into that architecture.
- `PREDICTA_CHART_INSIGHT_PHASE_4_KP_AND_NADI_CHART_MEANING_REBUILD` owns
  chart-meaning readability for KP/Nadi chart surfaces. This KP/Nadi phase
  expands that into full room UX, reports, PDF composition, and chat handoff.
- `PREDICTA_CHART_INSIGHT_PHASE_7_CHAT_REPORT_AND_CTA_INTEGRATION` remains the
  chart-to-chat/report integration gate. This KP/Nadi phase must feed that gate
  without duplicating it.
- `PREDICTA_LIFE_ATLAS_REPORT_STRICT_CONTRACT.md` remains the only approved
  all-school synthesis report. KP and Nadi may feed Life Atlas only as internal
  evidence layers there.
- `PREDICTA_NUMEROLOGY_PREDICTA_STRICT_PHASE.md` and
  `PREDICTA_SIGNATURE_PREDICTA_ULTRA_STRICT_PHASE.md` remain separate school

  reports.

  personality and response-style contract for Predicta. This phase adds

## Current Baseline Problems

The current foundations are useful, but not yet premium enough:

- KP web has event selection, cusps, significators, ruling planets, and proof
  path, but it exposes too much technical structure upfront.
- KP can feel like a table/proof panel instead of a calm event-answer room.
- Nadi web has story links, activations, validation questions, and source
  boundaries, but it can feel like many separate cards instead of one clear
  karmic story path.
- Mobile KP is stronger than mobile Nadi, but both need the same human-first
  hierarchy as web.
- Nadi mobile can feel like a card stack rather than a guided story.
- PDF KP/Nadi sections exist, but they read more like proof summaries than
  polished user-facing reports.
- Predicta memory must know more than the visible cards so she can explain the
  active KP question or Nadi story without guessing.

## Non-Negotiable Rules

1. KP Predicta and Nadi Predicta are separate specialist worlds.
2. KP reports must not become Vedic personality reports.
3. Nadi reports must not use KP cusp logic.
4. KP and Nadi reports must not silently include Numerology or Signature logic.
5. KP and Nadi may participate in `Predicta Life Atlas` only as clearly scoped
   internal evidence layers.
6. KP must be event-first, question-first, and timing-conscious.
7. KP must not guarantee an event, marriage, job, child, property, money, health
   result, court result, or irreversible life outcome.
8. KP must explain confidence and limitations clearly.
9. Nadi must be story-first, pattern-first, and validation-conscious.
10. Nadi must not claim real palm-leaf manuscript access unless Predicta has
    that actual source.
11. Nadi must not present karmic patterns as fixed punishment or unavoidable
    fate.
12. Technical evidence must exist, but it belongs behind a proof drawer on
    screen and in a `Proof Appendix` in reports.
13. The first user-facing layer must be plain-language meaning, not method
    jargon.
14. Free KP and Nadi must be useful, not hollow teasers.
15. Premium/paid KP and Nadi must add depth, proof, sequencing, timing, and
    practical guidance without fake certainty.
16. Web, mobile, reports, PDF output, and Predicta chat must use the same
    shared KP/Nadi intelligence contracts.
17. Missing or weak data must show an honest pending or low-confidence state,
    not fabricated certainty.
18. Every KP/Nadi report-visible insight must flow into Predicta memory/context
    through the app-wide memory architecture.
19. Predicta must remember the active KP event question and active Nadi story
    focus where the existing memory system supports it.
20. KP/Nadi user interfaces must avoid clutter. Use progressive disclosure,
    drawers, tabs, and compact summaries instead of long technical walls.

## Required KP Section UX

KP Predicta must become an `Event Answer Room`.

The first screen must show:

- `What are you asking?`
- simple event choices:
  - career
  - money
  - marriage
  - property
  - education
  - travel
  - custom event
- `KP's current answer`
- `Why Predicta is saying this`
- `When to watch`
- `What to do next`
- collapsed `Proof drawer`

The proof drawer may include:

- relevant houses
- KP cusps
- star lord
- sub lord
- sub-sub lord where available
- significator hierarchy
- ruling planets
- dasha support
- transit trigger windows
- confidence and limitations

KP must guide vague questions into better KP questions.

Example:

```text
KP works best with a specific event. Instead of "career?", ask "Will I change jobs in the next six months?"
```

## KP Wow Moments

KP must feel precise, useful, and calm.

### Event Verdict Compass

Show a polished compass with four axes:

- `Promise`
- `Block`
- `Timing`
- `Confidence`

This must summarize the answer without forcing the user into technical tables.

### Question-To-Proof Path

Show the user's event moving through a clear proof path:

```text
Event -> Houses -> Main Cusp -> Event Carriers -> Timing Support
```

This creates trust without clutter.

### Event Carriers Map

Show planets as:

- carriers
- supporters
- blockers
- timing triggers

Do not show only raw significator rows in the main view.

### Timing Readiness Bar

Show a simple timing state:

```text
Promise is visible. Timing support is partial. Watch for a stronger trigger before forcing the result.
```

### Ask Exact Question Wizard

Guide the user into a specific KP question with:

- event type
- desired time window
- current situation
- what outcome they are asking about

## Required Nadi Section UX

Nadi Predicta must become a `Karmic Story Room`.

The first screen must show:

- `Your strongest story thread`
- `The gift inside this pattern`
- `The repeating lesson`
- `Where this activates in life`
- `What Predicta needs to validate before going deeper`
- collapsed `Story evidence`

The main Nadi screen must feel like a guided path, not a pile of cards.

Preferred flow:

```text
Past Pattern -> Current Lesson -> Next Practice
```

Nadi validation must be simple and respectful:

- ask 3 to 5 clear questions
- do not make the user feel tested
- do not shame the user if validation is mixed
- use validation to deepen, soften, or correct the story

## Nadi Wow Moments

Nadi must feel intimate, reflective, and grounded.

### Karmic Story Map

Show a beautiful node map connecting:

- planets
- karakas
- life areas
- repeated themes
- activation windows

This map must be visual and calm, not mystical clutter.

### Rahu-Ketu Axis Card

Show:

- `What pulls you forward`
- `What you are learning to release`
- `Where the pattern becomes louder`
- `One practice for balance`

### Hidden Pattern Sentence

Generate one distilled story line.

Example:

```text
Your chart keeps turning pressure into self-mastery through responsibility, relationship mirrors, and delayed recognition.
```

### Validation Bridge

Ask 3 to 5 simple questions, then show:

- `Confirmed`
- `Partially confirmed`
- `Needs more validation`

Use this to improve the reading rather than pretending certainty.

### Activation Windows

Show when the story becomes louder without claiming guaranteed fate.

Approved framing:

```text
This pattern may become more noticeable during these periods.
```

Do not write:

```text
This event will definitely happen then.
```

### Room-Safe Behavior

Predicta must be able to say things like:

```text
You were looking at KP career timing. I will keep this event-focused and use the KP event carriers instead of turning this into a general Vedic reading.
```

Predicta must also be able to say:

```text
This is a Nadi-style karmic story reading, so I will treat it as a pattern to validate, not as a KP event judgement.
```

If the user asks for another school inside KP or Nadi, Predicta must hand off or
ask permission instead of silently mixing methods.

## Language And Tone Rules

KP language should be:

- crisp
- event-focused
- timing-aware
- practical
- confidence-labeled
- calm under anxiety

Nadi language should be:

- reflective
- pattern-aware
- intimate without fake authority
- karmic without fatalism
- validation-aware
- practical

Both rooms must avoid:

- fear language
- guaranteed predictions
- fake certainty
- technical overload in the first view
- generic spiritual filler
- method mixing
