# Predicta Email Support Inbox Resend Strict Phases

## Status

Approved controlling roadmap for the Predicta Email Experience System.

This roadmap exists because Predicta email must become a premium product
surface, not a utility bolted onto support forms.

This file is the controlling roadmap for:

- Resend outbound email delivery
- Resend inbound reply handling and webhook processing
- customer auto-replies
- admin notifications
- support ticket threading
- Outlook-like admin inbox
- admin reply composer
- premium email template system
- email localization readiness
- email security, privacy, and audit controls
- end-to-end email smoke testing

## Core Product Principle

Predicta emails must sound:

`Clear, warm, respectful, non-salesy, and action-oriented.`

Predicta emails must not sound like raw ticket software.

Bad:

```text
Ticket received. We will get back.
```

Correct:

```text
We have received your message and a Predicta team member will review it
carefully. If your request is about a report, payment, Kundli, or account
access, we will use the details you shared to help faster.
```

## Non-Negotiable Rules

1. Do not send Resend emails from client-side code.
2. Do not hardcode customer, admin, or system email templates directly inside
   random functions, route handlers, components, or screens.
3. Email templates must live in dedicated template source files.
4. Email copy must be localization-ready from day one.
5. Customer-facing templates and admin/internal templates must be separate.
6. Customer auto-replies must be warm, premium, and action-oriented.
7. Admin notifications must be operational, sortable, and direct.
8. Admin private notes must never appear in customer-visible emails.
9. Inbound email HTML must be sanitized before display.
10. Resend webhooks must be verified before any inbound message is trusted.
11. Outbound sends must be idempotent enough to avoid accidental duplicate
    customer emails.
12. Support ticket creation must survive email-provider failure.
13. Payment, refund, account-access, and premium-access email flows must be
    audited.
14. No astrology prediction may be sent in support email unless it was
    explicitly generated by an approved Predicta report/chat surface.
15. Email templates must include a short privacy/trust footer.
16. Admin inbox actions that change customer access, payment state, or report
    delivery must create an audit trail.
17. Do not expose internal routing details, private notes, provider IDs, raw
    webhook payloads, or staff-only metadata to customers.
18. No phase may be called green without local audit evidence.
19. No phase may be called green from code review alone.
20. Commit each green phase with a detailed commit message before moving to the
    next phase.

## Premium Email Design System

All customer-facing emails must use one premium Predicta email design system:

- dark Predicta header with subtle magenta, blue, and green atmosphere
- porcelain body background, not plain white
- deep navy text, not pure black everywhere
- restrained champagne dividers
- Predicta seal or logo shown small and clean
- one primary CTA button per email when a CTA is needed
- mobile-first body width around `600px`
- safe rendering in Gmail, Outlook, Apple Mail, and common mobile clients
- readable fallback text for clients that block images
- no giant decorative image that overpowers the message
- no narrow desktop-only layouts
- no tiny fonts, low contrast, or CTA crowding

## Admin Inbox Product Contract

The admin inbox must feel like a clean support desk, not a debug dashboard.

Required admin areas:

- inbox list
- ticket/thread detail
- customer context panel
- admin reply composer
- internal notes
- suggested templates
- status controls
- priority controls
- category controls
- audit trail

Required folders or filters:

- `Inbox`
- `Waiting on customer`
- `In review`
- `Resolved`
- `Escalated`
- `Billing`
- `Reports`
- `Kundli`
- `Signature`
- `Premium Access`
- `Bug`
- `Refund`

Admin must be able to:

- search tickets by customer, ticket number, email, category, and subject
- open a full thread
- select a suggested template based on ticket category
- edit the reply before sending
- add private internal notes
- send and mark as waiting
- send and resolve
- send and escalate
- see customer plan, account, report, and Kundli context when available
- see whether outbound email delivery succeeded or failed
- see whether inbound replies are attached to the correct thread

Admin must not be able to accidentally:

- send private notes to the customer
- change premium access without audited action
- reply from the wrong identity
- send duplicate confirmation emails
- expose raw provider metadata
- send unsupported astrology predictions through support templates

## Data Model Direction

The final implementation may use Firestore or the existing backend persistence
surface, but it must expose one canonical support data contract.

Required entities:

- `support_tickets`
- `support_ticket_messages`
- `support_ticket_events`
- `email_delivery_events`
- `email_templates`

`support_tickets` must track:

- ticket number
- status
- priority
- category
- subject
- customer identity
- source route
- language
- assigned admin
- latest message preview
- latest customer reply time
- latest admin reply time
- related Kundli/report/purchase identifiers when available

`support_ticket_messages` must distinguish:

- customer inbound message
- admin outbound reply
- system auto-reply
- internal private note

Private notes must be structurally impossible to send as customer email unless
an explicit conversion action is added later and audited.

## Phase 3

### Keyword

`PREDICTA_EMAIL_PHASE_3_RESEND_INBOUND_WEBHOOK_AND_THREADING`

### Goal

Accept inbound customer replies through Resend and attach them to the correct
support thread.

### Scope

- Add Resend inbound webhook endpoint at:
  `apps/web/app/api/email/resend/webhook/route.ts`.
- The production public webhook URL must be:
  `https://predicta.rudraix.com/api/email/resend/webhook`.
- Do not register the Firebase/App Hosting provider URL as the production
  Resend webhook unless the branded domain is unavailable and the fallback is
  explicitly approved.
- Configure the Resend webhook with these events:
  - `email.received`
  - `email.sent`
  - `email.delivered`
  - `email.delivery_delayed`
  - `email.bounced`
  - `email.failed`
  - `email.complained`
  - `email.suppressed`
- Do not enable `email.opened` or `email.clicked` in the first support-inbox
  implementation. They are not required for support threading and add privacy
  noise.
- Verify webhook signature or approved verification mechanism.
- Parse inbound sender, recipient, subject, body, HTML, text, and attachments
  only as allowed.
- Sanitize inbound HTML.
- Thread replies using provider metadata, reply-to address, ticket number,
  headers, or configured alias strategy.
- Record unthreaded inbound email safely for admin review.
- Add tests for duplicate webhook delivery.

### Must Touch

- webhook API route
- support message repository
- HTML sanitization helper
- inbound parsing tests

### Exact Execution Prompt

> Execute `PREDICTA_EMAIL_PHASE_3_RESEND_INBOUND_WEBHOOK_AND_THREADING`.
> Add verified Resend inbound webhook handling at
> `apps/web/app/api/email/resend/webhook/route.ts`, using the public production
> URL `https://predicta.rudraix.com/api/email/resend/webhook`. Subscribe to
> `email.received`, `email.sent`, `email.delivered`, `email.delivery_delayed`,
> `email.bounced`, `email.failed`, `email.complained`, and `email.suppressed`.
> Customer replies must attach to the correct support ticket thread when
> possible, and unthreaded messages must be safely quarantined for admin review.

### Strict Audit

- deployed/branded webhook URL is documented as
  `https://predicta.rudraix.com/api/email/resend/webhook`
- event subscription list excludes `email.opened` and `email.clicked`
- webhook verification tests pass
- sanitized HTML tests pass
- duplicate webhook idempotency tests pass
- known-ticket threading tests pass

## Final Green Definition

The Email Experience System is not green until:

- customer contact creates a durable support ticket
- customer auto-reply is sent from the server
- admin notification is sent from the server
- customer replies can be received and threaded
- admin can read and reply from a protected inbox
- admin can use premium templates
- private notes cannot leak
- templates are localization-ready
- payment/refund/access emails are audited
- Resend failures do not lose support tickets
- the UI is responsive and premium
- an end-to-end smoke is captured

Anything less is not a Predicta email system. It is just mail sending.
